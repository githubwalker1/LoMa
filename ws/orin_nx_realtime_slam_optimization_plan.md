# Orin NX 8GB 实时 SLAM 前端优化计划

## 1. 目标与边界

### 目标

在 Orin NX 8GB 上部署 LoMa 作为实时 SLAM 前端的局部特征模块。优化目标是单帧、单图对的稳定低延迟，而不是离线批量吞吐。

一帧处理包含：当前图像的特征点检测和描述子提取，以及与上一关键帧或局部地图的匹配。关键验收指标采用 P95 端到端延迟、峰值显存和 SLAM 跟踪质量。

### 初始部署约束

| 项目 | 约束 |
| --- | --- |
| 硬件 | NVIDIA Jetson Orin NX，8GB 统一内存 |
| 批次 | `B=1`，不通过凑 batch 增加排队延迟 |
| 输入 | `1 x 3 x 1024 x 1024` |
| 关键点数 | `N=2048` |
| 初始模型 | `loma_b128` |
| 初始精度 | FP16 |
| 模型形式 | `detector`、`descriptor`、`matcher` 三段式部署 |

`loma_b128` 不依赖 DINOv2 描述子，显存与时延风险最低，适合作为 Orin NX 8GB 的实时基线。`loma_b`、`loma_l`、`loma_g`、`loma_r` 在精度验证阶段再作为候选模型比较。

## 2. 性能预算与验收标准

在确定 SLAM 主循环频率前，先将目标延迟设为参数 `T_target`。若目标为 20 Hz，则单帧预算为 50 ms；若目标为 10 Hz，则单帧预算为 100 ms。应至少保留给状态估计、建图、回环和 ROS 2 通信的时间预算，LoMa 不应独占整帧预算。

| 指标 | 记录方式 | 验收原则 |
| --- | --- | --- |
| 延迟 | 连续 500 帧的 P50、P95、P99 | P95 小于分配给 LoMa 的帧预算 |
| 显存 | 模块级峰值和全进程峰值 | LoMa 建议不超过 4–5 GiB，至少保留约 2 GiB 系统余量 |
| 匹配质量 | 匹配数、MAGSAC 内点数、内点率 | 不低于基线的预设下限 |
| SLAM 质量 | 跟踪丢失次数、重定位成功率、ATE/RPE（有真值时） | 不低于基线的预设下限 |
| 稳定性 | 30 分钟以上连续运行 | 不出现 OOM、异常延迟尖峰或热降频 |

性能结论必须以同步后的 CUDA 计时为准，不能仅使用 Python 函数的墙钟时间。

## 3. 分阶段工作计划

### 阶段 0：准备可复现基线

1. 固定 50–100 组覆盖弱纹理、运动模糊、强曝光变化和大视角变化的图像对。
2. 固定输入预处理、图像尺寸、关键点数、模型版本、JetPack、TensorRT 和 CUDA 版本。
3. 在 Python F32 参考实现中记录关键点、描述子、匹配和 RANSAC 统计，作为数值正确性基准。
4. 在 Orin 上记录预热后的模块级 P50/P95、峰值显存、温度、功耗和 GPU 频率。

输出物：基线 CSV、测试图像对清单、当前模型配置和环境版本清单。

### 阶段 1：模块级计时与显存定位

按以下边界分别计时，并在每个阶段末尾用 CUDA Event 或等价机制同步：

1. 图像预处理与上传。
2. `detector`：稠密特征、NMS、top-k。
3. `descriptor`：稠密描述子网格和关键点采样。
4. `matcher`：Transformer 交叉注意力、相似度计算和匹配过滤。
5. CPU 后处理、坐标转换和 RANSAC。

检测和描述子均需对图像对中的两张图分别执行；Matcher 使用两组已缓存的描述子。应同时记录最大已分配显存、最大保留显存和 TensorRT workspace，避免将缓存分配器的保留内存误判为实时激活内存。

输出物：模块耗时占比表和显存瀑布图，明确第一优化目标。

### 阶段 2：导出与数值正确性验证

1. 使用 `scripts/export_loma_torchscript.py --kind split` 导出三段模型，不部署与 split 模型重复的 `detect_describe` 模型。
2. 保持单样本接口：图像为 `1 x 3 x 1024 x 1024`，关键点数为 2048。当前导出包装层即按该接口设计。
3. 依次验证 Python F32、TorchScript F32、TensorRT F32、TensorRT FP16 的输出。
4. 不要求不同后端的浮点值逐位一致；应比较关键点位置误差、描述子余弦相似度、有效匹配对重合率、MAGSAC 内点数和内点率。

建议门槛：FP16 结果相对于 F32 基线的内点率与 SLAM 跟踪成功率下降必须在预先约定的容差内；超出容差时停止性能优化并先定位算子精度问题。

### 阶段 3：固定 shape FP16 TensorRT 引擎

1. 基于板端 JetPack 版本选择可用转换链路，例如 Torch-TensorRT，或 ONNX 后由 TensorRT 构建引擎。
2. 先生成算子兼容性报告，重点检查 `grid_sample`、`topk`、NMS 相关操作、动态索引和注意力算子。
3. 仅为固定 shape 建立优化 profile：`B=1`、`H=W=1024`、`N=2048`，避免动态 shape profile 占用额外工作区和调度开销。
4. 三个模块分别构建与缓存 engine；记录 TensorRT layer profile，确认 descriptor 仍是主瓶颈。
5. 若某个算子无法下沉 TensorRT，优先评估局部插件或将轻量后处理移出引擎，不应把整个模块回退到 CPU。

预期效果：相对 Python/PyTorch F32，固定 shape FP16 TensorRT 的端到端延迟通常可降低约 35%–65%，整体显存可降低约 35%–55%。具体结果取决于 TensorRT 对该图中算子的融合程度。

### 阶段 4：运行时内存与数据路径优化

1. 预分配并复用输入、输出、临时缓冲和 TensorRT execution context，禁止逐帧创建 CUDA buffer。
2. 将 resize、归一化和色彩转换放至 GPU；若相机管线支持，使用 DMA-BUF/NVMM 或等价零拷贝路径，避免多次 CPU/GPU 复制。
3. 缓存上一关键帧或局部地图的关键点和描述子，只对当前帧运行 detector 与 descriptor。
4. 使用单一、有序的 GPU stream 保障低尾延迟；在确认资源充足后，再评估拷贝与计算的异步重叠。
5. 在进程启动时加载全部 engine 并完成预热，不在 SLAM 运行过程中进行模型加载、engine 构建或大块显存分配。

预期效果：相对已完成 FP16 TensorRT 的基线，运行时优化通常还能降低约 5%–15% 端到端延迟，并明显降低 P95/P99 抖动。

### 阶段 5：SLAM 集成与受控降载

若 FP16 固定 shape 仍不能满足 `T_target`，按以下顺序单变量调整，每一步均重复质量验收：

| 优先级 | 调整 | 预期性能影响 | 风险 |
| --- | --- | --- | --- |
| 1 | `1024 x 1024` 降至 `768 x 768` | 稠密计算量理论减少约 44%，端到端常见降低约 25%–40% | 小尺度结构和弱纹理点减少 |
| 2 | `N=2048` 降至 `N=1536` | Matcher 计算理论减少约 44%，端到端常见降低约 3%–10% | 匹配冗余和几何鲁棒性下降 |
| 3 | `N=1536` 降至 `N=1024` | Matcher 计算理论再明显下降 | 大视差、回环和低纹理场景风险增大 |
| 4 | 切换更快模型或调整关键帧策略 | 取决于实测 | 需要重新评估 SLAM 全链路质量 |

注意：减少关键点数主要降低 Matcher 负担，对稠密 descriptor 的主耗时帮助有限；若 descriptor 是瓶颈，应优先评估输入分辨率。

### 阶段 6：整机稳定性与功耗验证

1. 在获得明确授权后，将板卡切换到目标性能模式并锁定时钟，确保性能测试可复现。
2. 使用实际相机数据连续运行至少 30 分钟，监控温度、GPU 频率、功耗、P95/P99 延迟和内存增长。
3. 分别覆盖正常跟踪、频繁关键帧插入、重定位和回环等高负载场景。
4. 确认散热方案能够避免热降频后，再冻结部署参数。

输出物：整机稳定性报告、最终 engine 文件及版本信息、监控日志和可复现实验命令。

### 阶段 7：INT8 作为最后选项

只有在 FP16 和受控降载仍无法达到 `T_target` 时才进入 INT8：

1. 收集覆盖真实场景的代表性校准集。
2. 优先对兼容且收益明确的子图尝试 INT8，避免一开始将全部网络强制量化。
3. 对每次量化分别验证数值一致性、匹配内点指标和 SLAM 指标。
4. 若 INT8 导致跟踪质量越过容差，回退到 FP16，不以纯 FPS 作为发布依据。

预期效果：在 FP16 已优化的基础上，INT8 可能再带来约 10%–35% 的速度收益，但算子兼容性、实现复杂度和精度风险均最高。

## 4. 资源量级与风险控制

在 F32 下，1024 分辨率的稠密 detector 和 descriptor 会产生大量多尺度特征图；Orin NX 8GB 不适合作为实时部署精度。FP16 是默认部署前提。

| 模块 | 主要资源来源 | 优化策略 |
| --- | --- | --- |
| Detector | 高分辨率多尺度卷积特征图 | FP16、固定输入、GPU 预处理、避免重复运行 |
| Descriptor | 全图描述子网格和多尺度解码；部分模型还含 DINOv2 | 优先 FP16、控制分辨率、选择 `loma_b128` 基线 |
| Matcher | 点数平方级注意力和相似度矩阵 | 控制关键点数、复用关键帧描述子、保持固定 `N` |

风险门槛：若 LoMa 全进程峰值持续超过约 5 GiB、出现显存碎片化，或在稳定压力下发生热降频，应先处理内存与散热问题，而不是继续增加模型规模。

## 5. 最终交付物

1. 可复现的 Orin NX 性能基线与测试集清单。
2. 通过正确性验证的 FP16 TensorRT engines 和版本记录。
3. 模块级及端到端性能报告，包含 P50/P95/P99、显存、功耗和温度。
4. SLAM 集成验收报告，包含跟踪、重定位、回环与轨迹指标。
5. 已冻结的运行参数、回退方案和部署说明。

## 6. 决策原则

1. 先正确，后加速；任何优化不得绕过几何质量验证。
2. 先 FP16 固定 shape，再优化数据路径，最后才调整分辨率、点数和 INT8。
3. 以 P95 延迟和 SLAM 成功率为主要决策依据，不以单次最快延迟或离线吞吐替代。
4. 每次只修改一个变量，并保留可回退的 engine、配置和质量记录。
